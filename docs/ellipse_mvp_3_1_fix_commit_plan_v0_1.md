# Ellipse MVP 3.1 - рабочий план исправлений и коммитов v0.1

Дата: 2026-07-12
Статус: рабочий план, не GDD
Основание: `docs/ellipse_gdd_v3_1.md`, без правок GDD 3.1

## 0. Цель плана

Привести текущий Godot-прототип к ближайшему проверяемому состоянию MVP 3.1: читаемая арена, фиолетовая волна, дэш, safe gaps, дружественный синий источник, первый резонатор, активные эмиттеры и короткая последовательность обучающих боевых ситуаций.

План не пытается сразу собрать весь vertical slice. Хаб, граф локаций, полноценные NPC-сцены, предметная экономика и финальная структура Старого Шлюза остаются следующей фазой после боевого MVP.

## 1. Текущая база

Уже есть:

- Godot-проект в `godot/`;
- один боевой encounter в `godot/main.tscn`;
- игрок, движение, дэш, HP, i-frames;
- фиолетовые волны игрока по ЛКМ и каскад при удержании;
- красные круговые эмиттеры и золотой линейный эмиттер;
- стирание вражеских фронтов фиолетовой волной;
- красные danger nodes от пересечения вражеских волн;
- синие boost nodes от пересечения фиолетовых волн;
- победа после уничтожения 3 эмиттеров;
- Windows export preset и собранный билд.

Главные расхождения с MVP 3.1:

- боевые ассеты из `sprites_orig/` пока не подключены к Godot;
- арена процедурная, а не фоновая из ассетного пайплайна;
- нет отдельного синего дружественного источника / маяка;
- нет резонатора по ПКМ;
- нет room/encounter progression под обучение;
- нет boss-поведения Золотого Рыцаря, только золотой линейный эмиттер;
- нет отдельного слоя assist/readability-настроек;
- текущие untracked `build/` и export templates требуют решения: версионировать или игнорировать.

## 2. Правила коммитов

- Один коммит должен закрывать одну проверяемую ступень.
- GDD 3.1 не править в рамках этих коммитов.
- Документы планирования держать отдельно от GDD: `docs/*plan*`, `docs/*notes*`, `docs/*checklist*`.
- Сначала коммитить код и source assets, потом билд.
- Не смешивать балансные правки с рефакторингом.
- Не смешивать визуальный импорт ассетов с изменением правил боя.
- Каждый gameplay-коммит должен запускаться как отдельный playable build из редактора.

## 4. Что не делать в этой серии

- Не править `docs/ellipse_gdd_v3_1.md`.
- Не строить полноценный хаб.
- Не делать полный граф локаций.
- Не добавлять инвентарь, магазины и экономику.
- Не делать много резонаторных схем.
- Не вводить сложный boss bullet hell.
- Не делать процедурную генерацию.
- Не превращать резонатор в автономную турель.
- Не запекать волновую область, safe gaps, nodes или интерактивные источники в фон.

## 5. Минимальный Definition of Done для MVP 3.1

MVP 3.1 можно считать готовым к первому внешнему плейтесту, когда:

- игрок видит себя как персонажа, а не как абстрактную точку;
- красные фронты читаются как опасность;
- фиолетовая волна намеренно создает проход;
- safe gap воспринимается как маршрут;
- дэш используется для входа в проход;
- синий источник воспринимается как дружественный;
- ПКМ ставит резонатор, а ЛКМ запускает волну от игрока и резонатора;
- резонатор не воспринимается как автономная турель;
- активные и неактивные эмиттеры различимы;
- есть короткий flow из нескольких комнат и быстрый retry;
- есть Windows build или понятная инструкция запуска из Godot.
