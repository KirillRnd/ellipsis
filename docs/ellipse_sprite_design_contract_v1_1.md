# Эллипс — единый дизайн-контракт для спрайтов

**Версия:** 1.1  
**Назначение:** источник проекта для производства и интеграции персонажных спрайтов  
**Область:** визуальный слой `4.3`  
**Аудио:** не входит в этот документ

---

## 1. Главный принцип

Все персонажные спрайты должны производиться как части одной визуальной системы.

Для каждого персонажа сначала утверждается единый мастер-дизайн. Только после этого создаются:

- сценовый спрайт для интерлюдий;
- top-down-ракурсы;
- анимационные раскладки;
- отдельные предметные и световые слои.

Финальные кадры не генерируются как одиночные изображения. Для каждого смыслового положения создаются цельные банки вариантов с общим anchor, после чего лучшие кадры отбираются, нормализуются и собираются в runtime-sheet.

---

## 2. Производственная единица анимации

### 2.1. Один PNG — одно действие в runtime

Канонический финальный файл анимации:

> **персонаж × действие = один цельный нормализованный RGBA runtime-sheet**

Примеры:

```text
colorless_move_sheet.png
colorless_idle_sheet.png
rahn_move_sheet.png
rahn_action_sheet.png
```

Внутри одного runtime-sheet одновременно находятся:

- все отобранные уникальные ключевые фазы действия;
- единая камера;
- единый масштаб;
- единый дизайн персонажа;
- единый размер ячеек и pivot;
- прозрачный фон.

Порядок воспроизведения не обязан совпадать с физическим порядком ячеек. Игровой manifest может повторно использовать один и тот же ключевой кадр и задавать любую последовательность индексов.

Исходным материалом для отбора могут быть несколько chroma-key-банков. Каждый банк содержит несколько вариантов одного смыслового положения, а не готовый loop.

### 2.2. Почему генерируются банки поз

Генерация нескольких вариантов одной простой позы в общей раскладке помогает сохранить:

- лицо и пропорции;
- длину конечностей;
- одежду и её застёжки;
- расположение карманов, ремней и декоративных элементов;
- сторону асимметричных деталей;
- угол камеры;
- масштаб;
- толщину и характер линии;
- цвет и свет;
- узнаваемость персонажа внутри одного смыслового положения.

Одиночная генерация каждого финального кадра запрещена как основной пайплайн. Допускаются отдельные банки разных смысловых положений, если в каждом используется один утверждённый anchor и одинаковые инварианты персонажа.

### 2.3. Хромакей обрабатывается после отбора и до runtime-нарезки

Порядок:

```text
генерация chroma-key-банков ключевых поз
→ отбор кадров без обязательного сохранения исходного порядка
→ сборка chroma-key master-sheet действия
→ удаление хромакея со всего master-sheet
→ нормализация в единый RGBA runtime-sheet
→ техническая проверка
→ импорт в Godot
```

Хромакей удаляется одинаковым алгоритмом сразу со всей раскладки. Это обеспечивает одинаковый край альфа-канала на всех кадрах.

## 3. Формат runtime-sheet

После удаления хромакея исходник становится runtime-sheet:

```text
colorless_move_sheet.png
rahn_action_sheet.png
```

Runtime-sheet одновременно является:

- финальным графическим файлом действия;
- источником кадров для `SpriteFrames`;
- эталоном согласованности конкретной анимации.

Отдельные PNG-кадры могут автоматически нарезаться только для диагностики. Они не являются источником истины и не собираются вручную обратно.

### 3.1. Структура сетки

Runtime-sheet содержит одну строку. Столбцы задают уникальные ключевые фазы действия.

Фактическая последовательность, длительности, повторы и удерживаемые кадры задаются в manifest и игровой state machine. Один и тот же кадр разрешено использовать в последовательности несколько раз.

### 3.2. Один sheet на одно действие

Нельзя собирать все действия персонажа в одну гигантскую раскладку.

Правильно:

```text
colorless_idle_sheet.png
colorless_move_sheet.png
colorless_dash_sheet.png
colorless_place_resonator_sheet.png
colorless_crossbar_aim_sheet.png
colorless_crossbar_drive_sheet.png
colorless_hit_sheet.png
```

Так проще:

- повторно сгенерировать только неудачное действие;
- сохранить крупный размер ячеек;
- проверить движение;
- не смешивать разные постановочные задачи в одном изображении.

## 4. Контракт общей визуальной стилистики

### 4.1. Основные внутренние референсы

К общей группе приводятся новые версии Бесцветного, Крона и Рахна.

В качестве стилевых ориентиров используются:

```text
irvel.png
orum.png
tiu.png
warn.png
golden_knight.png
```

Они задают:

- рисованную техническую фэнтезийность;
- немного грубый, но контролируемый тёмный контур;
- ясные крупные формы;
- различимые материалы;
- умеренно условную анатомию;
- локальные цветные акценты;
- предметность одежды и инструментов;
- визуальную связь магии с устройствами и оптикой.

### 4.2. Референсы, требующие редизайна

Следующие файлы сохраняются как референсы идентичности, но не как финальные мастер-ассеты:

```text
colorless_v0.png
cron.png
player_bescolor_topdown_alpha_cropped.png
```

Они могут использоваться для сохранения:

- узнаваемых признаков;
- общей одежды;
- экипировки;
- цветовых акцентов;
- характера персонажа.

Их линия, рендер, пропорции и top-down-подача не считаются обязательным эталоном.

### 4.3. Общие запреты

Запрещены:

- гладкий 3D-рендер;
- фотореалистичная кожа и ткани;
- стерильная векторность;
- аниме-чистота;
- generic fantasy-стилизация;
- визуальная перегрузка мелкими деталями;
- детали, исчезающие после уменьшения до игрового масштаба;
- нарисованная постоянная тень под ногами, если тень создаётся в Godot.

Силуэт и крупные массы важнее внутреннего орнамента.

## 5. Контракт top-down-камеры

### 5.1. Камера

Используется единый строгий top-down-ракурс без набора рисованных направлений.

Точный угол фиксируется один раз по утверждённому эталонному sheet и затем не меняется во всём комплекте.

Обязательные свойства:

- камера неподвижна;
- горизонт отсутствует;
- видны голова, плечи, корпус и ноги;
- точка контакта с землёй читается;
- персонаж в исходном спрайте ориентирован к верхнему краю;
- перспектива и масштаб одинаковы во всех действиях;
- направление в игре задаётся вращением визуального узла целиком;
- зеркалирование кадров не используется, чтобы не менять стороны асимметричных деталей.

### 5.2. Базовая ориентация

Все исходные top-down-кадры направлены к верхнему краю изображения. Для движения, прицеливания и действий игровая логика вращает `VisualRoot` на произвольный угол.

## 6. Холст, сетка и pivot

### 6.1. Общий холст

Все ячейки внутри одного действия должны иметь одинаковый размер. Базовый размер сохраняется между действиями, кроме явно зафиксированных постановочных исключений.

Нельзя независимо tight-crop каждую ячейку или каждый кадр.

Для текущего комплекта Бесцветного используется базовая ячейка `256 × 256 px`.

Осознанное исключение — `dash`: его вертикально вытянутая постановка использует ячейку `256 × 512 px`. Этот sheet не уменьшается до базовой ячейки и не обрезается по высоте.

### 6.2. Pivot

Pivot фиксируется в одной и той же позиции каждой ячейки. Для текущего комплекта Бесцветного:

> `(128, 128)` — pivot базовой ячейки и середина шага между крайними положениями ступней

Для ячейки `dash` размером `256 × 512 px` используется pivot `(128, 384)`: дополнительные `256 px` находятся над общей точкой опоры и дают место вертикальному движению.

Требования:

- одинаковое положение pivot;
- одинаковый масштаб тела;
- одинаковая длина конечностей;
- отсутствие случайного вертикального дрожания;
- одинаковое положение персонажа относительно сетки;
- никакого автоматического смещения после удаления хромакея.

### 6.3. Безопасные поля

В каждой ячейке должен быть запас для:

- рук;
- плаща;
- длинной одежды;
- наклонов;
- рывка;
- постановочных поз.

Соседние кадры не должны пересекаться.

### 6.4. Сетка

- точная прямоугольная сетка;
- одинаковые размеры ячеек;
- отсутствие подписей и рамок внутри runtime-sheet;
- номера строк и столбцов накладываются только диагностическим скриптом;
- в исходном master-sheet фон — единый chroma-key-цвет, не встречающийся в персонаже;
- в runtime-sheet фон полностью прозрачный.

## 7. Master design sheet

Перед анимациями для каждого нового или перерабатываемого персонажа создаётся отдельный дизайн-лист.

Это одна полноценная картинка, а не runtime-анимация.

Минимальное содержимое:

- фронтальный вид;
- вид со спины;
- профиль;
- передний трёхчетвертной вид;
- задний трёхчетвертной вид;
- крупный фрагмент лица;
- крупные детали одежды и экипировки;
- палитра;
- материалы;
- фиксированные асимметричные элементы.

Имена:

```text
colorless_design_master_v1.png
cron_design_master.png
rahn_design_master.png
```

Design master является главным источником истины для всех дальнейших генераций.

## 8. Бесцветный

### 8.1. Что сохраняется

- серо-бесцветная основа;
- фиолетовые функциональные эффекты и предметные слои, но не постоянный цветовой акцент на теле;
- происхождение из Роя Обломков;
- практичная и собранная одежда;
- капюшонный или закрывающий голову силуэт;
- ощущение человека, использующего найденные предметы как инструменты;
- не рыцарь;
- не классический маг;
- не generic rogue.

### 8.2. Что меняется

Бесцветный приводится к той же общей стилистике, что:

```text
irvel.png
orum.png
tiu.png
warn.png
golden_knight.png
```

Нужно убрать ощущение персонажа из другой игры и чрезмерно цифровую детализацию.

### 8.3. Сценовый спрайт

```text
colorless_interlude_neutral.png
```

- прозрачный фон;
- не top-down;
- подходит для расположения преимущественно слева в интерлюдии;
- Стальная Поперечина и резонатор не обязаны быть навсегда встроены в руки;
- отдельные предметы могут накладываться отдельными слоями.

### 8.4. Top-down sheets

Базовый производственный набор:

| Действие | Что происходит | Важный визуальный акцент | Предмет | Режим |
|---|---|---|---|---|
| `idle` | Бесцветный переносит вес с одной стороны на другую и заметно дышит. | По металлическому наплечнику и линзе проходят спокойные переливы блеска. | Нет | loop |
| `move` | Бесцветный естественно и уверенно идёт вперёд, работая ногами, плечами и всем корпусом. | Движется вся фигура, а не только отдельно переставляемые ноги. | Нет | loop |
| `dash` | Бесцветный совершает короткий прыжок вперёд с места, без разбега. | Быстрый компактный отрыв и приземление. | Нет | one-shot |
| `hit` | Бесцветный резко пошатывается, а затем восстанавливает равновесие. | В момент удара по линзе проходит заметный резкий перелив. | Нет | one-shot |
| `place_resonator` | Бесцветный приседает к земле, протягивает руку, затем встаёт. | Действие читается по корпусу и руке без изображения устанавливаемого предмета. | Нет | one-shot |
| `crossbar_aim` | Бесцветный берёт брус в одну руку, целится, а после отпускания управления с усилием бросает его немного вперёд. | В одном действии есть удерживаемая фаза прицеливания и последующий бросок; финальный кадр возвращает к anchor. | Стальная Поперечина видна до броска и отсутствует после отделения | hold, затем one-shot |
| `crossbar_drive` | Бесцветный коротким движением работает брусом как лопатой и вдавливает его в землю. | Поза соответствует master design; движение короткое, без предварительного размаха. | Стальная Поперечина | one-shot |

Число уникальных ключевых кадров не фиксируется заранее. Оно определяется читаемостью действия и может быть меньше числа кадров воспроизведения, поскольку игровая последовательность вправе повторно использовать ключевые кадры.

### 8.5. Стальная Поперечина

Стальная Поперечина не рисуется внутри основных sheets тела.

Она существует отдельным PNG:

```text
steel_crossbar_topdown.png
steel_crossbar_interlude.png
```

Top-down-версия:

- стальной прут размером с обычный лом;
- прорезь ровно посередине;
- небольшой узор и оформленность;
- без массивной магической головки;
- без встроенного обязательного свечения;
- продольная ось горизонтальна в исходнике;
- pivot расположен в центре прорези.

В игре Поперечина:

- крепится к отдельному pivot;
- вращается независимо от тела;
- скрывается при переходе в мировой объект;
- создаётся отдельно в позиции Вбитого клина.

## 9. Крон

### 9.1. Что сохраняется

- принадлежность к Проводникам;
- синий цветовой язык;
- посох, маяк или инструмент Проводника;
- образ человека инфраструктуры, маршрутов и проходов;
- спокойная функциональная одежда;
- не маг-жрец, а Проводник.

### 9.2. Объём текущей версии

В рамках текущего слоя требуется только сценовый редизайн:

```text
cron_interlude_neutral.png
```

Top-down-анимации Крона сейчас не производятся.

## 10. Красный Погонщик Рахн

### 10.1. Визуальное направление

Основной внешний референс:

> Haradrim Soldier Wētā

Сохраняются:

- худой и высокий силуэт;
- вытянутые пропорции;
- человек из Красных;
- многослойные тёмно-красные, земляные и угольные ткани;
- закрытая или полузакрытая нижняя часть лица;
- сухая физическая сила;
- мобильность;
- отсутствие тяжёлой брони.

### 10.2. Запреты

Запрещены:

- рама для резонаторов на спине;
- сложные видимые механизмы;
- множество светящихся деталей;
- массивный тяжёлый силуэт;
- буквальное копирование костюма харадримов;
- слоновьи мотивы;
- перегруженность топ-даун спрайта.

### 10.3. Сценовый спрайт

```text
rahn_interlude_neutral.png
```

- прозрачный фон;
- не top-down;
- рассчитан на преимущественное размещение справа;
- должен быть тем же персонажем, что и top-down-версия.

### 10.4. Top-down sheets

| Действие | Ориентир уникальных кадров | Режим |
|---|---:|---|
| `anchor` | 1 | static |
| `move` | 4–8 | loop |
| `action` | 3–5 | one-shot |
| `defeat` | 1 | static |

`action` используется и для постановки резонатора, и для ручной команды синхронного залпа. Отдельные `idle` и `hit` для текущего runtime Рахна не производятся: покой показывает `anchor`, получение урона читается кратким программным flash.

Резонаторы не стреляют автоматически. Таймеры принадлежат самому Рахну.

### 10.5. Дополнительные слои Рахна

Отдельная яркая точка Рахна не используется. Получение урона и другие краткие сигналы создаются кодом и не требуют дополнительного PNG-слоя.

## 11. Другие игровые объекты

### 11.1. Фиолетовый резонатор

Готовая основа:

```text
resonator_crystal_base_alpha_cropped.png
```

Один и тот же спрайт используется для первого и второго резонатора игрока.

### 11.2. Красный резонатор

Для учебных комнат и боя используется обычный красный резонатор.

Отдельная «статическая учебная цель» не требуется.

Красный переносимый резонатор должен быть визуально родственен фиолетовому, но использовать красный цветовой язык.

Предлагаемое имя:

```text
resonator_red_base_alpha_cropped.png
```

Стационарный красный эмиттер:

```text
emitter_red_base_alpha_cropped.png
```

не считается автоматически переносимым красным резонатором.

## 12. Интеграция в Godot

### 12.1. Runtime

Каждый очищенный alpha-sheet импортируется целиком.

Для каждого действия создаётся одна базовая top-down-анимация в `SpriteFrames`.

Например:

```text
idle
move
dash
place_resonator
crossbar_aim
crossbar_drive
hit
```

### 12.2. Узлы

Рекомендуемая структура:

```text
CharacterBody2D
└── VisualRoot
    ├── Shadow
    ├── BodyAnimatedSprite2D
    ├── CrossbarPivot
    │   └── CrossbarSprite
    └── AnimationPlayer
```

Назначение:

- `AnimatedSprite2D` — покадровое тело;
- `SpriteFrames` — базовые top-down-анимации действий;
- `AnimationPlayer` — отдельные слои, события, свет, позиционирование предметов;
- игровая state machine — выбор действия, последовательности кадров и угла поворота `VisualRoot`.

`AnimationTree` не является обязательным для первого этапа.

### 12.3. События

События должны быть синхронизированы с кадрами:

- постановка резонатора;
- запуск волны;
- отпускание удерживаемой фазы прицеливания;
- отделение и бросок Поперечины;
- переход Поперечины в мировой объект;
- момент удара;
- получение урона.

События задаются в Godot, а не кодируются визуально в PNG.

## 13. Диагностическая нарезка

Из runtime-sheet разрешено автоматически создавать:

```text
audit/
  colorless_move_00.png
  colorless_move_01.png
  ...
```

Такая нарезка нужна для:

- поиска прыжков масштаба;
- сравнения лица;
- проверки одежды;
- поиска грязного альфа-края;
- проверки pivot;
- сравнения соседних фаз.

Диагностические кадры не редактируются вручную и не становятся новым мастер-источником.

## 14. Критерии приёмки

Sheet не принимается, если:

- персонаж меняет рост между кадрами;
- меняется длина конечностей;
- детали одежды мигрируют;
- левая и правая стороны меняются местами;
- меняется угол камеры;
- pivot плавает;
- тело дрожит без причины;
- цвета заметно гуляют;
- отдельный предмет не совпадает с позой рук;
- контур исчезает после уменьшения;
- альфа содержит ореол хромакея;
- в runtime-sheet присутствуют подписи или сетка;
- соседние ячейки пересекаются;
- кадры были независимо tight-cropped;
- анимация читается только крупно, но не в игровом масштабе;
- Бесцветный, Крон и Рахн выглядят как персонажи из разных художественных систем.

## 15. Итоговая формула пайплайна

### Design

```text
один master design sheet на персонажа
```

### Animation

```text
несколько chroma-key-банков смысловых поз
→ отбор уникальных ключевых кадров
→ один chroma-key master-sheet действия
```

### Processing

```text
удаление хромакея со всего master-sheet
→ нормализация масштаба и pivot
→ единый RGBA runtime-sheet с базовыми ячейками 256 × 256
→ осознанное исключение dash с ячейками 256 × 512
```

### Godot

```text
runtime-sheet + manifest последовательности
→ SpriteFrames
→ AnimatedSprite2D
```

### Separate layers

```text
Стальная Поперечина
тень
прочие световые и предметные слои
```

### Запрещённый путь

```text
одиночная генерация каждого финального кадра без anchor и банка вариантов
→ попытка сохранить исходный порядок генератора любой ценой
```

Такой путь не используется. Итоговый порядок кадров определяется смыслом движения и может свободно собираться из удачных кандидатов разных банков.
