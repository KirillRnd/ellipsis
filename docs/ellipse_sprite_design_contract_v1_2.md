# Эллипс — дизайн-контракт спрайтов и визуального runtime v1.2

**Дата:** 2026-07-18  
**Связанная версия:** GDD v4.5  
**Назначение:** самостоятельный контракт производства, подготовки и интеграции спрайтов, tutorial diagrams и ability HUD  
**Заменяет в текущем scope:** `ellipse_sprite_design_contract_v1_1.md`

---

## 1. Приоритет и область

Документ является источником истины для:

- персонажных runtime-sheets;
- прозрачности;
- размеров ячеек;
- pivot и регистрации;
- порядка кадров;
- отдельных предметных слоёв;
- interlude-спрайтов;
- пропорций объектов;
- tutorial diagrams;
- ability HUD;
- raw/runtime-границы репозитория.

Если этот контракт противоречит v1.1 в пределах Старого Шлюза, приоритет имеет v1.2.

Игровые назначения, числа механик и структура комнат находятся в `ellipse_gdd_v4_5.md`.

## 2. Главные принципы

1. Один персонаж имеет один согласованный дизайн.
2. Одно действие хранится отдельным runtime-sheet.
3. Все кадры действия имеют общий холст и pivot.
4. Порядок воспроизведения отделён от физического порядка уникальных кадров.
5. Прозрачность чистится до интеграции.
6. Runtime-файлы создаются отдельно от raw.
7. Спрайт никогда не растягивается вопреки исходным пропорциям.
8. Визуальный размер и anchor важнее формального совпадения bounding box.
9. Tutorial использует игровой визуальный язык, а не условные заменители.
10. Механический и видимый reach имеют один источник.

## 3. Repository policy

### 3.1. Raw

Защищённые каталоги:

```text
artbook/
sprites_orig/
```

Запрещено:

- редактировать файлы на месте;
- перезаписывать;
- переименовывать;
- менять crop;
- менять альфа-канал;
- массово коммитить внешний pack;
- использовать raw напрямую как runtime-зависимость Godot.

Разрешено:

- читать;
- измерять;
- анализировать контур;
- создавать новые производные файлы под `godot/assets`;
- добавлять воспроизводимые preparation scripts.

### 3.2. Runtime

Runtime хранится под:

```text
godot/assets/
```

Каждый runtime-файл:

- RGBA;
- имеет чистый прозрачный фон;
- не содержит checkerboard;
- не содержит цветного fringe;
- сохраняет детали по краю;
- имеет стабильное имя;
- импортируется Godot;
- может быть воспроизведён из raw скриптом.

## 4. Производственная единица

Каноническая единица:

> **персонаж × действие = один нормализованный runtime-sheet**

Примеры:

```text
colorless_move_sheet.png
colorless_dash_sheet.png
rahn_move_sheet.png
rahn_action_sheet.png
```

Один sheet содержит уникальные ключевые фазы. Повторы задаются manifest, `SpriteFrames` или кодом.

Запрещено:

- независимо генерировать каждый финальный кадр;
- вручную собирать кадры с разным масштабом;
- смешивать все действия в один гигантский sheet;
- использовать crop кадра как pivot.

## 5. Прозрачность и контур

Порядок подготовки:

```text
raw/chroma-key source
→ отбор
→ очистка прозрачности
→ анализ контура
→ регистрация по anchor
→ нормализация
→ runtime-sheet
→ Godot import
```

Контур используется для:

- поиска общей опорной формы;
- совмещения центра массы;
- контроля нижней точки;
- выявления случайных полупрозрачных пикселей;
- стабилизации кадра.

Только bounding box недостаточен: изменение жеста рук не должно двигать корпус.

## 6. Pivot и регистрация

### 6.1. Общий pivot

Pivot персонажа соответствует точке контакта с землёй или устойчивому центру тела.

Между кадрами запрещены:

- прыжок размера;
- дрожание;
- изменение визуальной высоты корпуса;
- смена центра из-за поднятых рук;
- поворот вокруг случайного края.

### 6.2. Anchor-frame

Для сложного набора выбирается anchor:

- нейтральная устойчивая поза;
- чистый контур;
- ясная нижняя точка;
- пригодность для сравнения со всеми действиями.

Каждый кадр регистрируется относительно anchor, а не предыдущего кадра. Это не позволяет ошибке накапливаться по цепочке.

### 6.3. Ключевые точки

При автоматической регистрации учитываются:

- нижняя точка корпуса;
- центр корпуса;
- центральная вертикаль;
- характерные внутренние массы;
- overlap контура.

Для Рахна регистрация обязана использовать contour overlap и внутренние признаки, а не только внешний crop.

## 7. Камера и ориентация

- top-down остаётся рисованным верхним трёхчетвертным ракурсом;
- свободное программное вращение допустимо только после визуальной проверки;
- направление движения и действия не должно быть обращено «наизнанку»;
- anchor, idle, dash, move, hit и place Resonator ориентируются в одну каноническую сторону;
- импортированный материал, направленный противоположно, разворачивается в runtime-копии;
- raw не изменяется.

## 8. Бесцветный

### 8.1. Runtime-файлы

```text
godot/assets/actors/colorless/colorless_idle_sheet.png
godot/assets/actors/colorless/colorless_move_sheet.png
godot/assets/actors/colorless/colorless_dash_sheet.png
godot/assets/actors/colorless/colorless_hit_sheet.png
godot/assets/actors/colorless/colorless_place_resonator_two_state_sheet.png
godot/assets/actors/colorless/colorless_crossbar_empty_hands_two_state_sheet.png
godot/assets/actors/colorless/colorless_defeat_sheet.png
godot/assets/interlude/characters/colorless.png
```

### 8.2. Idle

- нейтральная читаемая петля;
- масштаб совпадает с move;
- pivot совпадает с anchor;
- предметы не вшиты.

### 8.3. Move

Физические ключи:

- `Lshort`;
- `Llong`;
- `Rshort`;
- `Rlong`.

Playback:

```text
Lshort → Llong → Lshort → Rshort → Rlong → Rshort
```

Повтор коротких фаз задаётся playback, а не дублированными raw-кадрами.

### 8.4. Dash

- sheet сознательно имеет размер `256×512`;
- вертикальное поле необходимо вытянутому жесту;
- принудительное приведение к квадрату запрещено;
- playback: три кадра;
- скорость: `24 FPS`;
- покадровые offsets синхронизируют опорные точки с движением;
- визуальная фаза должна соответствовать `84 px` игрового перемещения.

### 8.5. Place Resonator

Два уникальных состояния:

```text
1 → 2 → 1
```

Спрайт Резонатора не вшивается в руки. Мировой объект создаётся отдельно.

### 8.6. Crossbar aim и drive

Источники пустых рук:

```text
colorless_crossbar_empty_hands_intermediate_01_alpha.png
colorless_crossbar_empty_hands_06_alpha.png
```

Playback:

```text
anchor → 1 → 2 → 1 → anchor
```

Aim и drive используют одно движение:

- две поднятые руки;
- aim останавливается в верхней точке;
- после отпускания продолжается drive;
- Ковырялка является отдельным слоем;
- world object отсутствует до impact.

### 8.7. Hit

- короткое действие;
- масштаб не меняется;
- invulnerability blink программный;
- отдельный световой baked effect не нужен.

### 8.8. Defeat

- используется `death.png` / подготовленный `colorless_defeat_sheet.png`;
- кадр сохраняет pivot текущей фигуры;
- defeat не заменяется исчезновением;
- после завершения runtime переходит в retry state.

## 9. Ковырялка

### 9.1. Три исходных назначения

```text
kovyryalka_driven_alpha_cropped.png
steel_crossbar_interlude_alpha_cropped.png
steel_crossbar_topdown_alpha_cropped.png
```

- `kovyryalka_driven` — вбитый вертикальный мировой объект;
- `steel_crossbar_interlude` — сценовое предметное изображение;
- `steel_crossbar_topdown` — горизонтальное top-down-представление с другой ролью.

Они не взаимозаменяемы.

### 9.2. Runtime

```text
godot/assets/items/kovyryalka_driven.png
godot/assets/items/steel_crossbar_interlude.png
godot/assets/items/steel_crossbar_topdown.png
```

### 9.3. Масштаб

- мировой объект масштабируется как часть арены, не как крупная иллюстрация;
- root сохраняет мировой scale `1`;
- scale изображения находится на дочернем visual;
- это позволяет рисовать reach в реальных пикселях;
- pivot world object соответствует точке контакта с Волной.

### 9.4. Иконки

HUD и tutorial используют:

```text
kovyryalka_driven.png
```

Иконка вписывается с сохранением aspect ratio.

## 10. Фиолетовый Резонатор

### 10.1. Источник и runtime

```text
sprites_orig/resonator_crystal_base.png
godot/assets/actors/resonator_crystal_base.png
```

Raw имеет квадратный холст с полями. Runtime корректно обрезан до:

```text
701×946
```

Это не квадратный объект.

### 10.2. Пропорции

Во всех местах размер вычисляется по высоте:

```text
width = height × source_width / source_height
```

Запрещено рисовать в квадрат:

- мировой Resonator;
- tutorial;
- HUD;
- interlude item;
- pickup preview.

### 10.3. Инстансирование

Один runtime PNG используется для первого и второго Резонатора. Отдельный второй спрайт не создаётся.

## 11. Рахн

### 11.1. Runtime

```text
godot/assets/actors/rahn/rahn_anchor.png
godot/assets/actors/rahn/rahn_move_sheet.png
godot/assets/actors/rahn/rahn_action_sheet.png
godot/assets/actors/rahn/rahn_defeat_sheet.png
godot/assets/interlude/characters/rahn.png
godot/assets/actors/resonator_red_base.png
```

### 11.2. Набор действий

| State | Кадры | Скорость |
|---|---:|---:|
| anchor | 1 | static |
| move | 4 | `6,5 FPS` |
| action | 5 | `8 FPS` |
| defeat | 1 | static |

Отдельных `idle` и `hit` нет.

### 11.3. Action

Action использует:

```text
anchor → action_2 → action_3 → action_2 → anchor
```

Это обеспечивает возвращение к опоре без скачка.

### 11.4. Запреты

- яркая отдельная точка Рахна;
- baked boss HP;
- случайное быстрое переключение sprite;
- смена размера между states;
- frame-to-frame crop alignment;
- отдельный idle, дублирующий anchor;
- отдельный hit-sheet.

Урон показывается программным flash, здоровье — программным HUD.

## 12. Крон и interlude

Крон в текущем scope имеет interlude-спрайт.

Interlude-контракт:

- фон из artbook подготавливается отдельно;
- фон не содержит встроенных runtime-персонажей;
- Бесцветный обычно слева;
- Крон или Рахн справа;
- спрайты прозрачны;
- размер стабилен между репликами;
- top-down assets не используются как портреты;
- Ковырялка может быть отдельным предметным слоем, если это требуется постановкой.

## 13. Gate и арена

### 13.1. Gate

- открытый и закрытый states;
- архитектурный масштаб;
- рисуется сразу поверх арены;
- не занимает полэкрана;
- боковые секции совпадают с коллизиями;
- визуальное состояние следует objective.

### 13.2. Фон

- без общего `0,82` opacity;
- без дополнительного затемнения `0,26`;
- без внутренней непрозрачной чёрной заливки;
- debug-рамка может оставаться чёрной;
- debug hitbox circle под Бесцветным не рисуется;
- gate не должен быть искусственно ярче или темнее фона из-за лишнего layer tint.

## 14. Tutorial diagrams

### 14.1. Общий принцип

Tutorial — статическая визуализация настоящей механики.

Используются:

- реальные персонажные спрайты;
- реальные Резонаторы;
- реальная Ковырялка;
- Kenney prompts;
- кодовая отрисовка Волн, gaps и узлов.

Не создаются gameplay Wave/Resonator nodes.

### 14.2. Волна

Tutorial повторяет runtime sandwich:

- glow: `crest_width × 2`, слабая альфа;
- line: `crest_width × 0,85`, насыщенная альфа;
- core: `2 px`, осветлённый цвет;
- базовая ширина `14 px`.

Абстрактная одноцветная дуга запрещена.

### 14.3. Погашение

- окружности действительно пересекаются;
- safe gap рисуется на обеих Волнах;
- half-angle берётся от игровой геометрии;
- светлая ширина `32 px`;
- спокойная середина `13,333333 px`;
- торцы `28 px`;
- suppress-color соответствует второй Волне;
- gap не тёмный.

### 14.4. Резонанс

Точки вычисляются как circle-circle intersections.

- две точки пересечения — два узла;
- С/Ф использует runtime blue/violet node;
- Ф/Ф использует runtime violet/violet node;
- запрещён узел в произвольном среднем месте.

### 14.5. Ковырялка

Карточка показывает:

- короткий ЛКМ;
- удержание ЛКМ;
- узкое Поглощение;
- широкое долгое Поглощение;
- иконку `kovyryalka_driven`.

## 15. Input prompts

Источник:

```text
sprites_orig/kenney_input-prompts_1.5
```

Лицензия: CC0.

Runtime allowlist:

```text
keyboard_w.png
keyboard_a.png
keyboard_s.png
keyboard_d.png
keyboard_space.png
keyboard_e.png
mouse_left.png
mouse_right.png
```

Правила:

- пакет целиком не копируется;
- prompt вписывается contain-алгоритмом;
- Space остаётся квадратным `64×64`;
- растяжение Space в широкий прямоугольник запрещено;
- prompt не перерисовывается вручную.

## 16. Ability HUD

Три квадратных слота:

- LMB / Ковырялка;
- RMB / Волна;
- E / Резонатор.

Состав:

- реальный input prompt;
- реальный предмет или кодовый фрагмент Волны;
- clockwise square fill;
- disabled tint;
- счётчик Резонаторов.

Все текстуры вписываются с сохранением aspect ratio.

## 17. Приёмка

### 17.1. Персонаж

- корпус не прыгает;
- нижняя точка стабильна;
- размер одинаков;
- альфа чистая;
- порядок кадров соответствует manifest;
- направление не вывернуто;
- action не мерцает.

### 17.2. Предмет

- роль файла соответствует назначению;
- мировой масштаб правдоподобен;
- reach совпадает с механикой;
- предмет не появляется раньше impact.

### 17.3. UI

- prompts не растянуты;
- Резонатор не квадратный;
- Wave sandwich совпадает с игрой;
- gaps видны на обеих Волнах;
- Resonance nodes находятся на пересечениях;
- cooldown заполняется по часовой стрелке.

### 17.4. Репозиторий

- raw не изменён;
- runtime и `.import` добавлены явно;
- внешний pack не staged целиком;
- preparation script воспроизводим;
- smoke test проходит.

## 18. Итоговая формула

```text
защищённый raw
→ очистка прозрачности
→ contour/anchor registration
→ нормализованный runtime-sheet
→ manifest playback
→ Godot
→ статическая tutorial-визуализация тем же языком
→ regression test
```

Нельзя исправлять визуальную проблему растяжением, случайным offset или ручной правкой raw. Исправляется источник runtime-геометрии, регистрация или contain-layout.
